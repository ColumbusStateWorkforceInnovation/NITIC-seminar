# ⚓ Admiral Bash's Lab TLS

How the lab cluster gets its HTTPS certificate.

## Production — real Let's Encrypt cert

The cluster on `wagbiz.org` serves a genuine **Let's Encrypt wildcard
certificate** for `wagbiz.org` + `*.wagbiz.org`, issued automatically by
**cert-manager** using the **Cloudflare DNS-01** solver. Set it up once on the
server:

```bash
just deploy-cert-manager     # installs cert-manager via Helm
just deploy-letsencrypt      # ClusterIssuer + wildcard Certificate
```

`deploy-letsencrypt` needs `CLOUDFLARE_API_TOKEN` (and `LAB_ADMIN_EMAIL`) set
in `lab.env`. cert-manager writes the issued cert into the **`wildcard-tls`**
secret in the `admin-tools` namespace, which the Gateway's HTTPS listener
(`k8s/core-tools/gateway.yaml`) consumes.

Because it is a real Let's Encrypt cert, **every browser, CLI tool, and the
Docker daemon trust it natively** — students do **not** install any CA.

Check it: `kubectl get certificate -n admin-tools`

## Offline fallback — self-signed cert

For a deployment with **no public DNS** (where the DNS-01 challenge can't run),
the repo keeps a self-signed path:

```bash
just cert         # generate-lab-cert.sh — self-signed CA + wildcard cert into certs/
just push-cert    # load certs/tls.crt into the wildcard-tls secret
```

In that mode **only**, each student VM must trust the lab CA:

```bash
INSTALL_LAB_CA=1 bash scripts/setup-client.sh <SERVER_IP>
```

(`setup-client.sh` skips the CA install by default — the production cert needs
no CA.)

## Files in this directory

| File | In Git? | Purpose |
|------|---------|---------|
| `ca.crt`  | yes | Self-signed CA — **fallback path only**; regenerated by `just cert` |
| `tls.crt` | yes | Self-signed wildcard cert — **fallback path only** |
| `ca.key`  | no (gitignored) | CA private key |
| `tls.key` | no (gitignored) | TLS private key |
| `README.md` | yes | This file |

> The committed `ca.crt` / `tls.crt` are ephemeral artifacts of the **fallback**
> path, not the production cert. On the Let's Encrypt deployment they are
> unused. Consider `git rm`-ing them (`just clean-certs`) and regenerating with
> `just cert` only if you ever need the offline fallback.
